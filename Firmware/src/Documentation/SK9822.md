# SK9822 Frame Builders

## Overview

`SK9822.c` provides functions that **build complete SPI frames** in the SK9822 wire format. The DMA driver (`SK9822_DMA_SSI.c`) shifts these frames out — see [SSI-DMA.md](SSI-DMA.md) for that side.

Every builder writes the same buffer layout:

```
┌─────────────┬───────────────────────────┬─────────────┐
│ START_BYTES │  NUM_LEDS × 4 bytes       │  END_BYTES  │
│  4 × 0x00   │  per LED:                 │  16 × 0xFF  │
│             │  [global][B][G][R]        │             │
└─────────────┴───────────────────────────┴─────────────┘
```

Total `FRAMESIZE` = **592 bytes** (4 + 143×4 + 16, supposed to be 144 but one led broke).

---

## SK9822 Wire Format

### Per-LED frame (4 bytes)

```
byte 0:  111 b b b b b     (top 3 bits = 0xE0, bottom 5 = brightness 0-31)
byte 1:  blue
byte 2:  green
byte 3:  red
```

The "global" brightness field is a 5-bit current scaler applied by the LED controller before its 8-bit PWM. Setting global = 0 turns the LED off without sending all-zero RGB, which is a useful distinction when chaining frames.

`START_BITS = 0xE0` is OR'd with the 5-bit brightness to produce the global byte. The driver passes `0xE0` (i.e. global = 0) as the global value for "off" pixels, and `START_BITS | brightness` for lit pixels.

### Wire order

The SK9822 datasheet specifies **BGR order** for the colour bytes, even though every builder accepts RGB inputs. The `WRITE_LED` macro reorders them on output:

```c
#define WRITE_LED(frame, idx, global, r, g, b) \
    do { (frame)[(idx)++] = (global);          \
         (frame)[(idx)++] = (b);               \
         (frame)[(idx)++] = (g);               \
         (frame)[(idx)++] = (r); } while(0)
```

Callers always pass natural `(r, g, b)`. The macro is the single point that knows about the wire order.

### Start and End

The 4-byte all-zero start is a synchronization marker — the SK9822 sees four 0x00 bytes and resets its internal frame state machine.

The end frame is `ceil(NUM_LEDS / 2)` bytes of **0xFF**. Its purpose is to keep the SPI clock running long enough for the last LED's data to propagate through the daisy chain. 16 bytes is enough in practice — the LEDs latch on idle clock, not on a specific bit pattern.

## Strip Geometry (POV Globe)

The blade has 143 active LEDs split between two faces:

```
Front face: LED 0  → LED 70  (71 LEDs, hub → tip)
Back  face: LED 71 → LED 142 (72 LEDs, tip → hub)
              ▲              ▲
              │              └ LED 142 = back hub, forced black
              └ LED 71 sits adjacent to LED 70 at the tip

Active LEDs per face = 71 (symmetric image across blade centre)
```

The chain length (`NUM_LEDS = 143`) is one short of the original 144 — the hub LED on the front face was physically removed and the back hub is forced black to keep the image symmetric.

```c
#define NUM_LEDS    143
#define FRONT_LEDS   71
```

## Compressed Format Builders

Different image data formats trade memory for colour fidelity. Every builder expands one column of compressed data into a full 592-byte frame.

| Builder | Bytes/LED | Colours | Per-column bytes | Use case |
|---------|-----------|---------|------------------|----------|
| `BuildFrame_1bit` | 1/8 | 2 + brightness LUT | 18 | Long animations, maximum compression |
| `BuildFrame_4bit` | 1/2 | 16 chosen | 72 | Animated GIFs (current default) |
| `BuildFrame_332`  | 1   | 256 fixed (RRRGGGBB) | 143 | Quick conversions, no palette generation |
| `BuildFrame_8bit` | 1   | 256 chosen | 143 | Static images (current default) |
| `BuildFrame_565`  | 2   | 65,536 | 286 | Highest quality, doesn't fit in SRAM for animation |

### 4-bit packing

Two LEDs per byte: high nibble = even index, low nibble = odd index. Since `NUM_LEDS` is odd, the last byte's low nibble is unused.

```c
uint8_t entry = (i & 1) ? (col4bit[i >> 1] & 0x0F) : (col4bit[i >> 1] >> 4);
```

### 8-bit palette

Direct array index into a `[256][3]` RGB palette. This is the simplest mapping and is what `BuildFrame_8bit` uses. The palette lives in flash; only the index column changes per frame.

### 565 / 332 expansion

```c
expand565(c, r, g, b)  /* RRRRRGGGGGGBBBBB → 8-bit channels */
expand332(c, r, g, b)  /* RRRGGGBB         → 8-bit channels */
```

These are bit-shift only — no palette lookup. Useful when you want to skip palette generation entirely or store data in a generic raster format.

## Special Builders

### `BuildFrame_Solid` — uniform colour

Used to clear frames at startup. Calls `NormalizeRGB()` first to scale the channels proportionally if `r + g + b > 255`. This preserves hue when the LEDs would otherwise clip to white.

### `BuildFrame_Pixel` — single LED

Lights one LED at position `pos`, leaves all others off (with global = 0xE0 = brightness 0). Used for hardware testing and bring-up.

### `BuildFrame_VU` — microphone VU meter

Lights LEDs from hub to tip on both faces, proportional to a level value [0, FRONT_LEDS]:

```
hub                                         tip
 ●─●─●─●─●─●─●─●─●─●─○─○─○─○─○─○─○─○─○─○─○      level = 10
green ────────────────► yellow ────► red
```

Both faces show the same bar height. The colour gradient spans hub→tip:

```c
uint8_t frac = (uint8_t)((uint32_t)i * 255 / FRONT_LEDS);
WRITE_LED(frame, idx, global, frac, 255 - frac, 0);
//  i = 0  → frac = 0   → (R=0,   G=255, B=0) green
//  i = 35 → frac = 125 → (R=125, G=130, B=0) yellow
//  i = 70 → frac = 251 → (R=251, G=4,   B=0) red
```

The back hub LED is always forced off to keep the image symmetric across the blade centre.

## Brightness Handling

There are **two** brightness controls:

1. **Per-LED 5-bit global** — the high 5 bits of the global byte, applied by the LED controller as a current limiter. Range 0–31.
2. **RGB values themselves** — the 8-bit PWM duty cycle for each channel.

The builders write both: `(global & 0x1F) | START_BITS` for the global byte, and the raw RGB values for the colour bytes. Most callers leave the global at a fixed value (e.g. 16) and modulate via the palette colours.

`NormalizeRGB()` only runs in `BuildFrame_Solid` and `BuildFrame_Pixel` because their callers might pass uncontrolled RGB values. Palette-based builders skip it because palettes are already designed with reasonable brightness levels.

## Public API

```c
void BuildFrame_Solid(uint8_t *frame, uint8_t r, g, b, brightness);
void BuildFrame_Pixel(uint8_t *frame, uint16_t pos, uint8_t r, g, b, brightness);

void BuildFrame_565 (uint8_t *frame, const uint16_t *col565,  uint8_t brightness);
void BuildFrame_332 (uint8_t *frame, const uint8_t  *col332,  uint8_t brightness);
void BuildFrame_1bit(uint8_t *frame, const uint8_t  *col1bit, uint8_t brightness);
void BuildFrame_4bit(uint8_t *frame, const uint8_t  *col4bit,
                     const uint8_t palette[16][3], uint8_t brightness);
void BuildFrame_8bit(uint8_t *frame, const uint8_t  *col8bit,
                     const uint8_t palette[256][3], uint8_t brightness);

void BuildFrame_VU  (uint8_t *frame, uint16_t level, uint8_t brightness);
```

All functions write exactly `FRAMESIZE` (592) bytes into `frame[]`.
